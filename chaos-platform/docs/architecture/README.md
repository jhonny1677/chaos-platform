# Architecture Overview

## What Problem Does This Platform Solve?

Software systems fail in unexpected ways. A database gets slow, a pod runs out of memory, a network partition splits the cluster in two. In production, these events catch teams off guard — services cascade, on-call engineers scramble, and customers see outages.

The Chaos Platform solves this by **failing your system on purpose, in a controlled way, before your customers do it for you**.

It has two main tools that work together:

1. **Chaos Engine** — deliberately breaks things inside your Kubernetes cluster (kills pods, adds network delay, exhausts CPU)
2. **Load Tester** — floods your services with realistic traffic to find the breaking point

Used together, you learn two critical things: *how does the system behave when something breaks?* and *at what traffic level does it start to degrade?*

---

## Who Would Use This in a Real Company?

**Site Reliability Engineers (SREs)** run chaos experiments as part of GameDays — planned exercises where the team deliberately triggers failures to verify that runbooks, alerts, and automatic recovery mechanisms work as expected.

**Platform Teams** use the load tester before a product launch to find the maximum safe traffic level and ensure auto-scaling kicks in before users feel it.

**Backend Engineers** use both tools during development to verify that their service handles partial failures gracefully — for example, that a timeout on a database call degrades the API gracefully rather than crashing the whole service.

**On-Call Engineers** use the weekly summary reports (generated by the Lambda function every Monday) to understand the platform's resilience trend over time and plan improvements.

## How the Two Main Tools Work Together

The chaos engine and load tester are designed to be used together, not in isolation.

**Step 1 — Baseline**: Run a load test at expected production traffic to capture your normal latency and error rate baseline. This becomes your "before" measurement.

**Step 2 — Chaos + Load**: While the load test is running at steady traffic, trigger a chaos experiment. This shows you exactly what happens to real user traffic when a component fails.

**Step 3 — Hypothesis Evaluation**: Before the experiment, you define a hypothesis: *"If we kill 50% of target-app pods, the error rate should stay below 5%."* After the experiment, the platform evaluates this automatically and tells you PASS or FAIL.

**Step 4 — Report**: A PDF report is automatically generated within seconds of the experiment completing and posted to Slack. It includes before/during/after metrics, a timeline of events, and specific recommendations if the hypothesis failed.

## Key Design Principles

**GitOps Everything**: All Kubernetes configuration is in git. ArgoCD watches the repo and applies changes automatically. There is no `kubectl apply` in the deployment process — only git commits.

**Observe First, Act Second**: Every component emits Prometheus metrics, structured JSON logs (shipped to Loki), and OpenTelemetry traces (shipped to Tempo). You should never be blind about what's happening.

**Least Privilege**: Every service has its own Vault policy, Kubernetes ServiceAccount, and IRSA role. A compromised chaos engine pod cannot read dashboard secrets, create new Vault tokens, or access other namespaces.

**Fail Safely**: The chaos engine has a hard 50% blast radius cap — it will never kill more than half the pods in a target deployment, preventing self-inflicted outages during experiments. A circuit breaker stops all further chaos if 3 consecutive experiments fail to recover.

## Component Map

| Component | Technology | Role |
|---|---|---|
| Target App | FastAPI (Python) | The system under test — fake e-commerce API |
| Chaos Engine | Python asyncio | Executes fault injection via Kubernetes API |
| Load Tester | Python + httpx | Generates HTTP traffic for baseline and chaos load tests |
| Dashboard | React 18 + Vite | Real-time UI for monitoring experiments and tests |
| Event Stream | Apache Kafka (MSK) | Chaos events stream between engine, scheduler, and observers |
| Live Stats Cache | Redis | Sub-second load test metric aggregation |
| Experiment Store | DynamoDB | Persistent record of all experiments and results |
| Report Storage | S3 | PDF reports, experiment result JSON, Terraform state |
| Report Generator | AWS Lambda | PDF generation and Slack notification on completion |
| Slack Notifier | AWS Lambda | Routes SNS events to appropriate Slack channels |
| Experiment Scheduler | AWS Lambda | Triggers experiments on EventBridge cron schedule |
| Observability | Prometheus + Grafana + Loki + Tempo | Metrics, dashboards, logs, traces |
| Secret Management | HashiCorp Vault | Dynamic credentials, policies, audit logging |
| Policy Enforcement | OPA + Kyverno | Admission control for all Kubernetes resources |
| Runtime Security | Falco | Syscall-level anomaly detection |
| TLS Certificates | cert-manager | Automatic certificate issuance and renewal |
| CD Pipeline | ArgoCD | GitOps-based continuous deployment |
| CI Pipeline | GitHub Actions | Build, scan, test, and push on every commit |
| Infrastructure | Terraform | AWS EKS, VPC, ECR, S3, DynamoDB, IAM |
